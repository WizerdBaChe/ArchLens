## 最新決策 (最新在前，按時間降序)

**決策ID**：DD-027  
**標題**：前後端生命週期分離與靜態檔案掛載發行機制  
**狀態**：Accepted  
**日期**：2026-06-13  
**決策者**：系統架構師  

### 情境  
為了降低最終使用者的安裝摩擦力，不能要求非前端開發者在機台上安裝 Node.js 與 npm。系統需區分「開發路徑」與「發行路徑」。

### 決策  
1. **發行路徑 (Production)**：前端透過 Vite 預先編譯（Build）為純靜態檔案（HTML/JS/CSS），並複製至 `archlens_backend/static/`。後端 FastAPI 掛載該靜態目錄，直接於根路由 `/` 提供網頁服務。
2. **自動化腳本 (Automation)**：建立 `setup.bat` (環境初始化)、`dev.bat` (平行啟動開發伺服器)、`build.bat` (靜態打包與搬移) 與 `run.bat` (終端使用者純 Python 啟動腳本)。

### 後果  
正面影響：達到最小摩擦力的軟體交付，使用者僅需 Python 即可運行完整的全端應用程式。
負面成本：開發者在發布新版本前，必須記得執行 `build.bat` 進行靜態檔打包。

**決策ID**：DD-026  
**標題**：右側預覽區實作純前端高效檔案匯出機制 (Client-side Structural Exporting)  
**狀態**：Accepted  
**日期** 2026-06-13  
**決策者**：系統架構師  

### 情境  
使用者在前端控制樹調整完畢並動態生成 ASCII 樹狀圖後，需要將結果保存為本地實體檔案（如用於 README.md 中）。若透過後端進行寫檔與下載，將產生不必要的網路傳輸開銷與伺服器硬碟 I/O 負擔。

### 決策  
完全脫離後端依賴，在前端 `TreeView.tsx` 組件中直接利用瀏覽器的 `Blob` 與 `URL.createObjectURL` API 實作純前端下載邏輯。  
提供雙模匯出選擇：
1. **純文字模式（.txt）**：直接匯出純 ASCII 樹狀文字。
2. **Markdown 模式（.md）**：自動在外層包裹 Markdown 代碼區塊語法（```），方便使用者直接整合至專案文件。

### 後果  
正面影響：實現 0 延遲的極速下載體驗，完全不消耗後端資源；架構完全解耦，未來前端部署到靜態主機（如 Vercel）時該功能依然完美運作。
負面成本：無。

**決策ID**：DD-025  
**標題**：實作全局折疊控制與語意豁免機制 (Global Toggle & Semantic Exemption)  
**狀態**：Accepted  
**日期**：2026-06-13  
**決策者**：系統架構師  

### 情境  
先前實作的「智慧折疊演算法 (DD-022)」採用一刀切的策略，可能導致重要源碼檔案（如大量 `.ts` 或 `.py` 檔）被誤收合，且使用者缺乏強制展開全結構的控制權。

### 決策  
1. **API 合約擴充**：於 `FormatRequest` 新增 `enable_truncation` 布林值，並在前端 UI 提供全局切換按鈕。
2. **語意豁免 (局部開放) 機制**：於 `formatter.py` 內建一組「原始碼白名單（EXEMPT_EXTENSIONS）」。若檔案屬於原始碼或重要設定檔，則無視數量上限，強制全數渲染；折疊演算法僅針對非白名單的雜訊檔案生效。

### 後果  
正面影響：把控制權還給使用者，並透過白名單機制在「畫面清爽」與「資訊完整」之間取得完美平衡。
負面成本：需維護一份副檔名白名單清單。

**標題**：實作遞迴 UI 預設收合與 ZIP 串流分析介面

### 情境

3. **ZIP 上傳**：在 `App.tsx` 建立隱藏式 File Input，透過 `ArchLensAPI.scanZipFile` 將實體檔案傳送至後端暫存分析。

**決策ID**：DD-022  
**標題**：渲染層實作同類型檔案智慧折疊演算法 (Smart Truncation)  
**狀態**：Accepted  
**日期**：2026-06-12  
**決策者**：系統架構師  

### 情境  
實務資料夾內常含有大量同類型檔案（如數十個 `.dll`, `.csv` 或圖像檔），這會導致 ASCII 樹狀圖被同質性資訊洗版，失去結構導覽的意義。

### 決策  
在 `services/formatter.py` 中抽取獨立的 `truncate_and_sort_children` 核心處理方法。將資料夾與檔案分離解耦，檔案部分依副檔名群組。同一目錄下若相同副檔名的檔案超過 3 個，僅渲染字母排序前 3 個，其餘動態生成一個不帶大小資訊的虛擬省略提示節點（如 `... (還有 x 個 .dll 檔案)`）進行收合輸出。

### 後果  
正面影響：極大化 ASCII 結構圖的可讀性與清爽度，防止 README 因暫存/編譯檔破版。
負面成本：破壞了原始目錄的完整 1:1 對照（但此為 README 架構說明圖的預期功能最佳化）。

**決策ID**：DD-021  
**標題**：掃描層動態整合目標專案之 .gitignore 規則  
**狀態**：Accepted  
**日期**：2026-06-12  
**決策者**：系統架構師  

### 情境  
不同框架（如 Vite, Django）會產生各自的編譯檔與暫存檔，靜態的 `config.json` 難以涵蓋所有隨時間推移產生的全新邊緣情境，導致掃描結果冗長。

### 決策  
維持 `config.json` 作為底線防禦。同時在 `services/scanner.py` 中新增 `parse_gitignore` 方法。在遞迴走訪到各級子目錄時，若偵測到該目錄下存在實體 `.gitignore` 檔案，則即時解析該規則，並在該層的遞迴範疇（Scope）內局部疊加忽略清單，實現高度動態防禦。

### 後果  
正面影響： ArchLens 能自動無痛相容各種不同語言、不同開發框架的專案排除邏輯，不需使用者手動配置。
負面成本：遞迴過程中頻繁進行 `os.path.exists` 與檔案讀取，微幅增加硬碟 I/O 負擔（可透過提早剪枝機制抵消）。

**決策ID**：DD-020  
**標題**：前端狀態管理採用頂層集中式架構 (Top-down State Management)  
**狀態**：Accepted  
**日期**：2026-06-12  
**決策者**：系統架構師  

### 情境  
實作「拼圖化互動樹狀圖」需要管理一棵龐大的 JSON 狀態樹（包含 `is_enabled` 的切換）。需要決定是否引入外部狀態管理庫（如 Zustand, Redux），或是利用 React 原生的 Context API，亦或最單純的頂層 State 集中管理。

### 決策  
考量到後端已具備強大的提早剪枝機制（Ignore List），前端樹狀節點數量受控，因此不引入任何外部狀態套件。所有 Node 狀態統一儲存於頂層 `App.tsx` 中，透過 Props 向下傳遞給子元件，並由頂層提供一個遞迴更新函式來處理勾選狀態的變更。

### 後果  
正面影響：架構極致輕量化，符合 MVP 精神；無須學習額外套件 API，未來交由 AI 維護或重構的門檻極低。
負面成本：當專案樹狀節點意外突破數千個時，頂層狀態更新會觸發全畫面重新渲染（Re-render），可能產生微幅的 UI 延遲。

**決策ID**：DD-019  
**標題**：前端技術棧升級為 TypeScript 以賦能 AI 與強化合約  
**狀態**：Accepted  
**日期**：2026-06-12  
**決策者**：系統架構師  

### 情境  
前端專案即將初始化。若採用原生 JavaScript，在處理複雜的樹狀資料結構與後端 API 拋轉時，缺乏嚴格的型別約束，極易發生「屬性名稱打錯卻不報錯」的靜默失敗（Silent Failure）。此外，考量到未來高度仰賴 AI 生成 UI 程式碼，AI 需要極度清晰的上下文脈絡。

### 決策  
將前端建置目標從 `react` 升級為 `react-ts` (TypeScript)。
利用 TypeScript 的 `interface` 功能，在前端 1:1 完美映射後端 Pydantic 的 `NodeSchema`。

### 後果  
正面影響：
- 達成「防禦性程式設計」，前後端合約被編譯器強制保護。
- AI (如 Copilot/Gemini) 在撰寫或重構元件時，擁有 100% 精準的屬性自動完成 (Autocomplete) 提示，幾乎消滅低級語法 Bug。
負面成本：需額外維護 `tsconfig.json` 型別設定檔與 `interface` 定義，程式碼稍微增加幾行型別宣告。

**決策ID**：DD-018  
**標題**：前端引入 Axios 封裝網路層並配置 Vite 透明代理  
**狀態**：Accepted  
**日期**：2026-06-12  
**決策者**：系統架構師  

### 情境  
隨著前後端分離與 Monorepo 架構確立，前端（localhost:5173）與後端（localhost:8000）處於不同網域。為了解決跨網域（CORS）限制、避免在前端程式碼中寫死後端 IP 網址，並確保未來 AI 擴充 API 時有統一的錯誤處理與請求封裝機制，需要鎖定前端的通訊協定工具與代理規則。

### 決策  
1. **網路請求庫**：選用 `axios` 作為前端 HTTP 客戶端，拋棄原生 `fetch`，以利用其強大的轉換與攔截器特性。
2. **開發期透明代理**：於 `vite.config.js` 中配置 `server.proxy` 規則，將所有指向前端 `/api` 的請求，在後端自動代理轉發至 `http://127.0.0.1:8000`。
3. **介面合約模組化**：建立統一的 `src/api/client.js`，將後端提供的 Pydantic Schemas 映射為 JavaScript 的異步函式物件，前端元件不允許直接呼叫 raw axios。

### 後果  
正面影響：
- 完美隔離環境變數，前端發送請求一律使用相對路徑（如 `/api/v1/...`），未來上線部署時前端程式碼完全不需修改。
- 透過 Axios 攔截器可實作全域錯誤捕獲（例如後端崩潰時自動彈出提示）。
負面成本：前端專案正式增加了 `axios` 套件依賴。

**決策ID**：DD-016  
**標題**：採用 Monorepo 雙資料夾結構進行前後端物理隔離  
**狀態**：Accepted  
**日期**：2026-06-12  
**決策者**：系統架構師  

### 情境  
隨著架構由 CLI 演進為 Web 服務（FastAPI + React），後端的 Python 環境生態（Pip/venv）與前端的 JavaScript 環境生態（npm/Node）將在專案中並存。若將兩者混在單一根目錄下，會導致配置文件打結、追蹤混亂，且不利於 AI 專注於單一技術棧的重寫任務。

### 決策  
將 `archlens` 專案總目錄重構為 Monorepo 雙子目錄架構：
1. **頂層目錄**：僅保留全專案的 `.gitignore`、`.idea` 配置與 `ADR.md`。
2. **`archlens_backend/`**：存放 FastAPI、Schemas、以及移入 `services/` 子目錄的核心解析與渲染模組（`scanner.py`, `formatter.py`）。
3. **`archlens_frontend/`**：存放獨立的 Vite + React + Tailwind 前端專案。

### 後果  
正面影響：
- 實現真正意義上的架構解耦，前後端專案各自擁有獨立的配置與工具鏈。
- 大幅優化 AI 協作體驗，可單獨將前端資料夾交付給前端 AI 進行 UI 改造，隔絕無關上下文。
負面成本：本地開發時需要同時在兩個子目錄下分別啟動 Uvicorn 伺服器與 Vite 開發伺服器。

**決策ID**：DD-015  
**標題**：FastAPI 後端架構採用 Pydantic Schema 與 Router 分離設計  
**狀態**：Accepted  
**日期**：2026-06-12  
**決策者**：系統架構師  

### 情境  
進入 Web 後端實作階段，API 層必須具備高度的可擴充性，以應付未來潛在的新功能（如動態配置傳遞、多檔案上傳、使用者狀態管理），同時需要嚴格驗證前後端傳遞的 JSON 資料結構。

### 決策  
1. **依賴注入與資料驗證**：全面採用 `pydantic` 建立 Request / Response 模型（Schemas）。將我們在 DD-005 定義的 OOP Node 映射為 `NodeSchema`，確保進出 API 的資料格式絕對安全。
2. **路由解耦**：使用 `APIRouter` 並加上 `/api/v1/` 前綴。未來若 API 規格大改，可直接開設 `/api/v2/` 保持向下相容。
3. **CORS 配置**：預先開啟 CORS (Cross-Origin Resource Sharing) 中介軟體，允許未來在不同 Port（如 Vite 的 5173）運行的前端順利呼叫。

### 後果  
正面影響：極致的模組化，API 文件（Swagger UI）會由 FastAPI 自動精確生成，前端（或 AI）可直接對照文件刻畫面。
負面成本：專案目錄結構會增加幾個特定的資料夾（如 `routers`, `schemas`），初期檔案數量微幅上升。

**決策ID**：DD-014  
**標題**：互動介面採用狀態保留與半透明剪枝無視圖 (Rich Tree with Active-State Pruning View)  
**狀態**：Accepted  
**日期**：2026-06-12  
**決策者**：系統架構師  

### 情境  
為了提供極致便利的「拼圖化操作體驗」，使用者需要直觀地調整最終 README 樹狀圖的呈現內容。若直接將節點從 UI 畫面上刪除，使用者將無法找回或重新啟用該節點。我們需要一種既能保持全域架構視角，又能動態過濾輸出的設計。

### 決策  
將網頁介面拆分為兩個完全解耦的區塊：
1. **控制區（全狀態樹）**：永遠渲染完整的 `Node` 結構。每個節點綁定一個 `isEnabled` 的布林狀態。當狀態為 `false` 時，UI 透過 CSS 將該節點與其子樹調整為「半透明（Opacity-40）」狀態，代表其已被隱藏，但節點依然保留在畫面上供隨時重新啟用。
2. **預覽區（動態輸出視圖）**：獨立於控制區之外。當任何節點的 `isEnabled` 改變時，系統會動態將「僅包含 Active 節點」的過濾資料結構送入 `formatter.py` 演算法，即時重新渲染出乾淨的 ASCII 文字與中繼資料，並在專屬的 Code Block 內呈現。

### 後果  
正面影響：提供極具安全感且直觀的互動體驗，使用者可任意「開關」節點而不遺失上下文；完全對齊了前後端分離與無狀態渲染的演算法哲學。

**決策ID**：DD-004  
**標題**：過濾機制採用 fnmatch 標準函式庫實作萬用字元支援  
**狀態**：Accepted  
**日期**：2026-06-12  
**決策者**：開發團隊  

### 情境  
在前置分析中發現，實務專案常包含大量同類型的忽略檔（如數十個 `.dll`、`.so` 或 `*.log`），且資料夾可能帶有特定前綴（如 `test_*`）。若 `scanner.py` 僅採用純字串比對（`==` 或 `.endswith()`），配置檔將變得極度冗長且難以維護，無法滿足「大量編譯檔與動態連結庫」的邊界測試基準。我們需要一個支援萬用字元（Wildcard）的解析方案，同時又不能違背「不依賴外部套件（MVP 篩選原則）」的設計準則。

### 決策  
1. **技術選型**：採用 Python 內建的 `fnmatch` 模組（具體為 `fnmatch.fnmatch(filename, pattern)`）來實作 Ignore List 的比對邏輯。
2. **配置檔格式約定**：外部的 `config.json` 或 `.ignore` 檔案中，允許使用者輸入如 `*.dll`, `__pycache__`, `build/*` 等標準 Unix shell 風格的萬用字元。
3. **模組內實作細節**：在 `scanner.py` 的目錄走訪迴圈（如 `os.walk` 或 `pathlib.Path.rglob`）中，加入攔截層：讀取到任何檔案或目錄名稱時，迭代 Ignore List 進行 `fnmatch` 比對。一旦命中（Return True），若是檔案則跳過紀錄；若是目錄，則使用 `os.walk` 的 `dirs.remove(d)` 技巧原地修改（In-place modification）或在遞迴中提早 Return，以避免浪費 I/O 資源去深度走訪注定被捨棄的子目錄。

### 後果  
正面影響：
- 完美遵守不使用第三方套件的原則。
- 提供等同於 `.gitignore` 的基礎彈性，大幅提升使用者配置的靈活性。
- 提早剪枝（Pruning）被忽略的目錄，降低深層巢狀結構的 I/O 負擔與記憶體消耗。

負面成本：
- `fnmatch` 底層是轉換為 Regex 執行，在單一目錄下擁有十萬級別檔案數量且 Ignore List 極長的情境下，CPU 負擔會微幅上升（但考量此為 I/O 密集型工具，此 CPU 運算成本完全可接受）。
- `fnmatch` 不完全等同於 `gitignore` 的進階語法（例如不支援 `**` 跨目錄層級匹配），需在未來的文件中向使用者明確說明其語法邊界。

正面影響：極致的低耦合與高可攜性，核心邏輯能無痛移植到 FastAPI 等 Web 後端。
負面成本：需自行實作部分（原本可由第三方套件提供的）邊界處理邏輯。
